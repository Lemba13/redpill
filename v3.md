# RedPill ‚Äî Vertical 1: Autonomous Research Planning

> Replace static query lists with a two-stage, reasoning-LLM-powered research planner that decomposes the topic, learns from its own discoveries, and evolves its search strategy across runs.

## Key design decisions

- **Two LLM clients**: The research planner uses `qwen3.5:4b` via Ollama (reasoning model) for topic decomposition. Everything else (summarization, term extraction) stays on the existing local model (e.g. `llama3.1:8b` or `qwen3:4b`).
- **Two-stage planning**: Stage 1 is LLM-powered decomposition (what dimensions of this topic should I explore?). Stage 2 is deterministic query synthesis (convert each dimension into search queries). This separation means the LLM does the hard thinking and the code does the mechanical query construction.
- **Research plan persists across runs**: The plan is stored in SQLite and updated each run, not regenerated from scratch. The LLM sees yesterday's plan plus today's new findings and refines it.
- **Fallback is always deterministic**: If the reasoning LLM fails, the system falls back to a simple term-based expansion strategy. The pipeline never breaks.

## Phase 1 ‚Äî Database Schema Extensions (state.py)

```sql
CREATE TABLE IF NOT EXISTS query_log (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    query_text    TEXT NOT NULL,
    run_date      DATE NOT NULL,
    source        TEXT NOT NULL,  -- "llm_planned", "fallback", "static", "base_topic"
    reasoning     TEXT,           -- LLM's reasoning for this query (if applicable)
    results_count INTEGER DEFAULT 0,  -- raw results from Tavily
    new_items     INTEGER DEFAULT 0,  -- results that passed dedup
    kept_items    INTEGER DEFAULT 0   -- results that made it to final digest
);
```

### 1.3 Add `research_plans` table

```sql
CREATE TABLE IF NOT EXISTS research_plans (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    topic         TEXT NOT NULL,
    run_date      DATE NOT NULL,
    plan_json     TEXT NOT NULL,       -- full research plan as JSON
    reasoning_trace TEXT,              -- the reasoning model's chain of thought (if available)
    source        TEXT NOT NULL DEFAULT 'llm'  -- "llm" or "fallback"
);
```

### 1.4 Add DB helper methods to `StateDB` class

- `store_extracted_terms(terms: list[dict])` ‚Äî upsert terms; if a term already exists, increment `frequency` and update `last_seen`
- `get_recent_terms(days: int = 14) -> list[dict]` ‚Äî terms with `last_seen` within N days, ordered by frequency desc
- `get_top_terms(limit: int = 20) -> list[dict]` ‚Äî all-time top terms by frequency
- `log_query(query_text: str, run_date: str, source: str, reasoning: str = None)` ‚Äî insert a query log entry
- `update_query_stats(query_text: str, run_date: str, results_count: int, new_items: int, kept_items: int)` ‚Äî update counts after search/dedup/delivery
- `get_query_performance(days: int = 14) -> list[dict]` ‚Äî query log entries from last N days with all stats
- `save_research_plan(topic: str, run_date: str, plan: dict, reasoning_trace: str = None, source: str = "llm")` ‚Äî store today's plan
- `get_latest_research_plan(topic: str) -> dict | None` ‚Äî get the most recent plan for a topic

### 2.1 Add a second LLM client config

In `config.yaml`, add a `planner_llm` block alongside the existing `ollama_config`:

```yaml

ollama_config:
  base_url: "http://localhost:11434"
  model: "qwen3:4b"

# NEW ‚Äî used only for the research planner (once per run)
planner_llm:
  base_url: "http://localhost:11434"
  model: "qwen3.5:4b"
  think: true  # enable reasoning/thinking mode
```

Both use Ollama, but different models. The planner model is a reasoning model that supports `think: true`.

### 2.2 Update LLMClient or create a PlannerLLMClient

The planner LLM call should:
- Use `qwen3.5:4b` model
- Enable thinking mode (`think: true` in the Ollama API request)
- Request JSON output (`format: "json"` in the Ollama API request)
- Have a longer timeout than normal calls (reasoning can take 30-60 seconds on a 4b model)
- Capture and return the thinking/reasoning trace separately from the final answer

Ollama API call for thinking mode:
```python
response = ollama.chat(
    model="qwen3.5:4b",
    messages=[{"role": "user", "content": prompt}],
    format="json",
    think=True,  # enables extended thinking
    options={"num_ctx": 8192}  # generous context for planning
)
# response.message.content = final JSON answer
# response.message.thinking = reasoning trace (if think=True)
```

## Phase 3 ‚Äî Term Extractor (term_extractor.py)

### 3.1 Create `redpill/term_extractor.py`

This module uses the **existing local LLM** (not the reasoning model) to extract domain-specific terms from articles after summarization.

### 3.2 Implement `extract_terms(content: str, title: str, url: str, topic: str, llm_client: LLMClient) -> list[dict]`

Prompt the LLM with the article content and ask it to extract:
- **Subtopics**: specific areas within the main topic (e.g. for "contrastive learning" ‚Üí "self-supervised vision", "hard negative mining")
- **Techniques/methods**: named algorithms, architectures, approaches (e.g. "SimCLR", "MoCo v3", "BYOL")
- **Key authors/labs**: researchers or groups driving work in this space
- **Datasets**: benchmark datasets mentioned (e.g. "ImageNet-1K", "CIFAR-100")
- **Frameworks/tools**: software or libraries relevant to the topic (e.g. "lightly.ai", "solo-learn")
- **Emerging keywords**: any other novel or trending terms in context

Each extracted term should include: `term`, `category`, and `relevance` (1-5 score for how central the term is to the base topic).

Filter out generic terms that are too broad to be useful as search queries (e.g. "machine learning", "neural network", "deep learning" for an ML topic).

Return format: `[{"term": "...", "category": "...", "relevance": 3, "source_url": "...", "source_title": "..."}]`

### 3.3 Implement `extract_terms_batch(items: list[dict], topic: str, llm_client: LLMClient) -> list[dict]`

- Takes all of today's digest items (each has `content`, `url`, `title`)
- Calls `extract_terms` for each item
- Deduplicates extracted terms across items (same term from multiple articles ‚Üí keep highest relevance, note frequency)
- Returns the merged, deduplicated list

### 3.4 Error handling

- Wrap `json.loads()` in try/except. If Ollama returns malformed JSON, strip markdown backticks before retrying parse. If still fails, log a warning and return empty list for that item.
- Use Ollama's `format: "json"` to reduce JSON failures.
- The pipeline must never crash due to term extraction failure.

### 3.5 Prompt design for term extraction

The prompt should:
- Request JSON-only output explicitly: "Respond with valid JSON only. No preamble, no markdown, no explanation outside the JSON."
- Provide the base topic so the LLM understands what's "specific" vs "generic"
- Include one concrete example of good output
- Cap the output at 10-15 terms per article to avoid noise

## Phase 4 ‚Äî Research Planner (query_planner.py)

This is the core intelligence of the system. It has two stages.

### 4.1 Create `redpill/query_planner.py`

### 4.2 Stage 1: Topic Decomposition ‚Äî `decompose_topic()`

```python
def decompose_topic(
    topic: str,
    db: StateDB,
    planner_llm: PlannerLLMClient,
    max_dimensions: int = 6
) -> dict:
```

This function calls the **reasoning LLM** (`qwen3.5:4b` with `think: true`) to produce a structured research plan.

**Inputs to the prompt:**
1. The base topic from config
2. The previous research plan from `db.get_latest_research_plan()` (or empty if first run)
3. Recent extracted terms (last 14 days) via `db.get_recent_terms()`
4. Top terms overall via `db.get_top_terms()`
5. Recent query performance via `db.get_query_performance()` ‚Äî which queries yielded results that survived dedup
6. Today's date ‚Äî for temporal awareness (conferences, seasonal trends)

**What the LLM should produce (JSON):**

```json
{
  "dimensions": [
    {
      "name": "hard negative mining strategies",
      "description": "Methods for selecting informative negative samples",
      "priority": "high",
      "coverage": "under-explored",
      "suggested_queries": ["hard negative mining contrastive", "negative sampling strategies 2026"]
    },
    {
      "name": "self-supervised vision benchmarks",
      "description": "Latest benchmark results and comparisons",
      "priority": "medium",
      "coverage": "well-covered",
      "suggested_queries": ["self-supervised learning benchmark results"]
    }
  ],
  "dropped_dimensions": [
    {
      "name": "basic SimCLR tutorials",
      "reason": "Well-covered in prior runs, low novelty expected"
    }
  ],
  "new_directions": [
    "Noticed emerging interest in contrastive learning for audio ‚Äî worth a probe query"
  ]
}
```

**Prompt design ‚Äî this is the most critical prompt in the system:**

```
You are a research strategist planning the next search run for a daily research digest.

Topic: {topic}
Today's date: {date}

Previous research plan (from last run):
{previous_plan_json or "No previous plan ‚Äî this is the first run."}

Domain terms extracted from recent articles (last 14 days):
{recent_terms_formatted}

All-time top terms:
{top_terms_formatted}

Query performance from recent runs:
{query_performance_formatted ‚Äî show which queries found content that made it to the digest vs which found nothing new}

Your task: Produce an updated research plan with 4-6 dimensions (subtopics/angles) to investigate.

For each dimension:
- "name": short label (3-6 words)
- "description": one sentence explaining what this covers
- "priority": "high" | "medium" | "low" ‚Äî based on how much new content is likely to exist
- "coverage": "under-explored" | "partially-covered" | "well-covered" ‚Äî based on what prior runs have already found
- "suggested_queries": 1-2 realistic web search strings (3-8 words each, the kind you'd type into Google)

Also include:
- "dropped_dimensions": any areas from the previous plan you're removing, and why
- "new_directions": any emerging angles suggested by recent findings that weren't in the previous plan

Rules:
- Always keep one dimension for the base topic itself (broad catch-all)
- Always keep at least one "exploratory" dimension probing an adjacent area
- Deprioritize dimensions where recent queries found zero new content
- Prioritize dimensions where extracted terms are trending (appearing with increasing frequency)
- Search queries should be realistic web search strings, not academic jargon dumps
- Good query: "contrastive learning hard negatives 2026" ‚Äî specific, likely to find content
- Bad query: "contrastive self-supervised representation learning methodologies" ‚Äî too dense, won't match real articles

Respond with valid JSON only. No preamble, no markdown, no explanation outside the JSON.
```

**Capture the reasoning trace:** When `think: true` is enabled, `qwen3.5:4b` produces a thinking trace before the final JSON. Store this trace in `research_plans.reasoning_trace` ‚Äî it's invaluable for debugging why the planner made certain decisions.

### 4.3 Stage 2: Query Synthesis ‚Äî `synthesize_queries()`

```python
def synthesize_queries(
    plan: dict,
    topic: str,
    max_queries: int = 5
) -> list[dict]:
```

This is **deterministic** ‚Äî no LLM needed. It converts the research plan into concrete search queries:

1. Query 1 is always the base topic verbatim (reliability anchor, hard-coded regardless of plan)
2. Remaining query budget: pull `suggested_queries` from plan dimensions, prioritized by: `high` + `under-explored` first, then `medium`, then `low` or `well-covered` last
3. Always include at least one query from a `new_directions` item if present
4. Cap at `max_queries` total

Return format:
```python
[
    {"query": "contrastive learning", "source": "base_topic", "reasoning": "Base topic anchor"},
    {"query": "hard negative mining contrastive 2026", "source": "llm_planned", "reasoning": "High-priority, under-explored dimension"},
    {"query": "contrastive learning audio modality", "source": "llm_planned", "reasoning": "New direction: emerging cross-modal interest"},
    ...
]
```

### 4.4 Orchestrator function ‚Äî `plan_queries()`

```python
def plan_queries(
    topic: str,
    db: StateDB,
    planner_llm: PlannerLLMClient,
    max_queries: int = 5
) -> list[dict]:
```

This is the top-level function called from `main.py`:

1. Call `decompose_topic()` to get the research plan
2. Save the plan to DB via `db.save_research_plan()`
3. Call `synthesize_queries()` to convert plan ‚Üí queries
4. Log the full query plan at INFO level
5. Return the query list

If `decompose_topic()` fails (LLM error, timeout, malformed JSON), catch the exception, log a WARNING, and call `plan_queries_fallback()` instead.

### 4.5 Fallback ‚Äî `plan_queries_fallback()`

```python
def plan_queries_fallback(
    topic: str,
    db: StateDB,
    max_queries: int = 5
) -> list[dict]:
```

Deterministic, no LLM needed. The safety net.

1. Query 1: base topic (e.g. "contrastive learning")
2. Query 2: base topic + current year (e.g. "contrastive learning 2026")
3. Queries 3-N: pick top extracted terms by frequency from `db.get_top_terms()` and combine each with the base topic (e.g. "contrastive learning SimCLR", "contrastive learning hard negatives")
4. If no extracted terms exist yet (first run), use the static `search_queries` from config (backward compatible with MVP)

Return same format as `plan_queries` with `source: "fallback"`.

## Phase 5 ‚Äî Pipeline Integration (main.py, search.py)

### 5.1 Modify `search.py`

- Change `search_topic(topic: str, ...)` to `search_queries(queries: list[dict], ...)` ‚Äî accepts a list of query dicts instead of a single topic string
- Run each query through Tavily, collecting results
- Tag each result with which query produced it (for the query log)
- Deduplicate URLs across queries (same URL from two different queries ‚Üí keep it, attribute to the first query that found it)
- Return results with an extra field: `source_query` linking back to the query text that found it

### 5.2 Modify `main.py` orchestrator `run_pipeline()`

Updated pipeline flow:

1. Load config and initialize DB (existing)
2. **NEW** ‚Äî Initialize planner LLM client from `config.planner_llm`
3. **NEW** ‚Äî Run query planner: call `plan_queries()` to generate today's queries
4. **NEW** ‚Äî Log each planned query to `query_log` table
5. Run search with dynamic queries (modified ‚Äî uses new `search_queries()`)
6. Extract content from URLs (existing)
7. Filter through dedup engine (existing)
8. **NEW** ‚Äî After dedup, update `query_log` with `results_count` and `new_items` per query
9. Summarize new items and generate digest (existing)
10. **NEW** ‚Äî After summarization, run term extraction on today's items: call `extract_terms_batch()`, then `db.store_extracted_terms()`
11. **NEW** ‚Äî Update `query_log` with final `kept_items` counts
12. Deliver digest (existing)
13. Update state DB with all new items (existing)

### 5.3 Backward compatibility

- If `query_planning.enabled` is `false` or missing in config, fall back to the original static query behavior from MVP (use `search_queries` list from config directly)
- The pipeline must still be idempotent ‚Äî running twice in a day should not double-extract terms or double-log queries (check by `run_date`)

## Notes for Claude Code

- This builds on the existing RedPill project ‚Äî do NOT restructure existing modules, only extend them
- The two new modules (`term_extractor.py`, `query_planner.py`) should follow the same patterns as existing modules: use the `LLMClient` abstraction, use `logging`, use type hints
- All LLM prompts should request **JSON-only output**: "Respond with valid JSON only. No preamble, no markdown, no explanation outside the JSON."
- For Ollama JSON parsing: always wrap `json.loads()` in try/except. If the model wraps JSON in markdown backticks, strip them before parsing. This is a known Ollama quirk. Prefer using `format: "json"` in the Ollama API call to reduce failures.
- The decomposition prompt (Phase 4.2) is the most important prompt in the system ‚Äî it should be clear, well-structured, and include concrete examples of good vs bad output
- The planner uses `qwen3.5:4b` with `think: true` ‚Äî this is a different model and call pattern than the summarization LLM. Keep them as separate client instances.
- Store the reasoning trace from `qwen3.5:4b` ‚Äî it's available via `response.message.thinking` when `think=True` is passed. Log it and save it to the `research_plans` table.
- Keep `plan_queries_fallback` simple and deterministic ‚Äî it's the safety net when the reasoning model is slow or returns garbage
- The pipeline must remain robust: if planning fails, fall back. If term extraction fails for one article, skip it and continue. Never crash the pipeline for a non-critical step.
- When in doubt about a design decision, optimize for simplicity over cleverness
